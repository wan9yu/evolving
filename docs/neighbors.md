# Neighbors — where `ev` sits

Decisions deserve infrastructure, and a growing set of tools is built on that premise. This page places `ev` among them honestly: what each neighbour is for, how it works, and where `ev` takes a different path.

`ev` combines three properties:

1. an **immutable, content-addressed chain** of decisions,
2. the **grounds** a decision rests on as first-class structured data, and
3. a **falsifiable check** bound to a ground that, when it goes red, **actively resurfaces** the decision.

Properties (1) and (2) are shared with several neighbours below. Property (3) — *re-evaluating a recorded decision against a changing world and bringing it back when its premise breaks* — is the one `ev` leads with. But (3) is no longer rare: a shipping tool (Engram, below) now surfaces decision conflict *and* checks memory against the codebase. So the sharper line is a **second axis — *how* a decision is re-evaluated**:

> `ev`'s trigger is a **deterministic, falsifiable check**. It surfaces a **fact** — a test went red against the project's real git state — and **refuses to arbitrate**: a human ratifies and decides. The rest of the category takes the opposite path — probabilistic and automatic: an **LLM judges** the conflict, a **confidence score** drifts, the agent **self-judges and auto-resolves**.

That axis — *facts over verdicts, surface over auto-resolve, a human on the hook* — is where the differences below now concentrate, more than the bare idea of resurfacing.

## The neighbours

| Neighbour | What it is for | How it works | `ev`'s different path |
|---|---|---|---|
| **ADR tools** — log4brains, npryce/adr-tools, MADR | Write and publish architecture decisions | Static markdown; supersession by a "superseded" status, by convention | The record is content-addressed and carries a runnable check — a broken premise brings the decision back, rather than relying on someone re-reading a file |
| **adr-kit** — [rvdbreemen/adr-kit](https://github.com/rvdbreemen/adr-kit) (its *Guardian*) | Keep ADRs from going stale for AI coding agents | A scheduled staleness sweep (session-start + a weekly CI cron that maintains one "ADR guardian audit" issue); detection is heuristic — regex *Enforcement* rules against the diff, plus age / supersession / policy-drift ranking | `ev` runs an **executed** falsifiable check that goes red on a real failure, tied to the decision's premise — not a scheduled heuristic. (adr-kit's proactive timer + auto-issue is machinery `ev` deliberately lacks — `ev` is pull-only.) |
| **Lore Protocol** ([arXiv:2603.15566](https://arxiv.org/abs/2603.15566)) | Structured knowledge for AI coding agents, in git commit trailers | Grounds-style trailers (constraints, rejected alternatives); staleness via five passive heuristics (age, code drift, confidence, expiry hint, orphaned deps), surfaced on demand with `lore stale` | Binds a *falsifiable* check that is re-run and goes red on a real failure — an executed verdict, not a heuristic staleness flag |
| **haft** — [m0n0x41d/haft](https://github.com/m0n0x41d/haft) | An engineering-decisions engine "that knows when decisions are stale" | A trust **score** (`R_eff`) that decays as evidence ages, plus `valid_until` dates; a measure step reopens a decision when a prediction fails or the score crosses a threshold | The reopen-on-failed-measurement instinct is shared, but `ev` deliberately has **no score or decay** (facts-not-verdicts): its trigger is a **binary** falsifiable check going red (gating one git hook), not a numeric trust threshold |
| **AgDR** — [me2resh/agent-decision-record](https://github.com/me2resh/agent-decision-record) | Record what an agent decided | A record format for agent decisions | `ev` watches whether the decision is *still true*, and inverts the authorship line — a human ratifies and stays on the hook |
| **Decision ledgers** — agentic-decision-ledger, ElixirData | Govern decisions at decision time | Pre-execution admissibility gates; ElixirData adds a live decision stream and a manual "replay" of a past decision | `ev` does not gate at write time; it re-checks a *recorded* decision later, automatically, when the world it rested on moves |
| **Agent memory** — Mem0, Zep/Graphiti, Letta/MemGPT, Hindsight | Remember facts and context across sessions; some now calibrate (Hindsight "reflects") | Passive recall; staleness, where handled, is an **LLM-judged confidence drift** (Hindsight's `Assess` nudges a stored opinion's confidence on contradiction) | `ev` is active *and deterministic*: a falsifiable check goes red — a fact — not a confidence score an LLM adjusts. Judgment and governance, not recall. (LongMemEval, the memory benchmark, measures conversational recall, not decision-drift — a different problem) |
| **Engram** — [Gentleman-Programming/engram](https://github.com/Gentleman-Programming/engram) | Agent memory that surfaces decision conflict **and** checks memory against the codebase | FTS5 finds candidate conflicts → an **LLM judges** the relation (`conflicts_with` / `supersedes` / …); a periodic scan compares stored facts to config / manifests / Dockerfiles and **auto-resolves** stale memory against them | The closest shipping tool on these two axes — but `ev` inverts the mechanism: a **deterministic falsifiable check** (any author-written test, git- and cross-diff-aware) over a config-value scraper; it surfaces a **red fact and lets a human decide** rather than LLM-judging and auto-resolving; and a human **ratifies** (no agent self-judging). `Thezenmonster/agentmem` is a nearer *governance* analog (deterministic conflict + a decision lifecycle), still without a runnable check or a ratify gate |
| **Signed-provenance / agent identity** — content-addressed + signed memory protocols (BLAKE3 + Ed25519), agent-identity schemes | Cryptographic provenance and capability boundaries for agent memory | Signed chains; identity and authorization | `ev` is honest-by-construction — provenance is *declared, not cryptographic*; integrity is *tamper-evident* (`ev verify` detects edits), not write-enforced. None of these watch a decision's premise |
| **Architecture-fitness / policy-as-code** — ArchUnit, OPA, fitness functions | Enforce architectural rules in CI | Block the build on a violation | `ev`'s default posture is to *resurface*, not block; blocking is opt-in (`--exit-on-red`) and scoped to the git hook — and it ties the check back to the *past decision and its grounds*, not just "a rule was violated" |
| **Safety-assurance cases** — GSN runtime-assurance ([arXiv:2511.14805](https://arxiv.org/abs/2511.14805)) | Argue a safety case stays valid at runtime | A `<<RuntimeAssumptionMonitor>>` on a goal node that, on a violation, marks the node `<<Reopened>>` — the same monitor-on-a-recorded-assumption → reopen shape as `ev`'s (3) | The *concept* matches, but it operates on safety-case **goal nodes** in a modelling notation (the shipped artifact is a design-time plugin), not human **decisions** in a git-native chain. `ev` is that idea *implemented*, *decision-scoped*, and *git-native* |
| **Governance frameworks** — NIST AI RMF (MEASURE 2.4 / 3.1), EU AI Act Art. 12 | Standards and regulation for AI monitoring and logging | Frameworks and mandates | `ev` is a single-binary substrate that can help satisfy a continuous-monitoring outcome (e.g. NIST MEASURE 2.4/3.1) — advisory alignment, not a compliance product |

## Honest edges

- **Shared ground — and it is narrowing, honestly.** An immutable chain and first-class grounds are not unique to `ev` (Lore has both). And *active resurface* is no longer just a concept: **Engram ships it** — decision-conflict surfacing plus a codebase ground-truth scan. So `ev`'s claim is now narrower and more precise. It is **not** "we resurface" (others do); it is the **mechanism and posture**: a **deterministic falsifiable check** (an executed test, not an LLM verdict or a confidence score), **surfaced not auto-resolved** (`ev` refuses to arbitrate), **human-ratified** (no agent self-judging), on a **git-native, decision-scoped, immutable chain**. Each of those is shared by *someone*; the **combination — and the refusal to be probabilistic or automatic — is `ev`'s ground.**
- **`ev`'s own gaps.** Provenance is **declared, not cryptographic** — signing is a deliberate non-goal, not a shipped feature. Immutability is **tamper-evident** (`ev verify` flags any edit), **not** write-enforced; `ev` is advisory, never a gate beyond a git hook. It fires on git-recorded change, never on external-state drift that leaves no commit.
- **No universal claim.** "Not found elsewhere" means exactly that — a survey, not a proof of a vacancy. If a tool already watches a decision's premise and resurfaces it, that is a neighbour we have not yet met, and we would want to know.
